# AGENTS.md

Operational field manual for agents working on jalki.

Read `CLAUDE.md` first for the full architecture. Jalki is a programmable
fentry/fexit framework for kernel and runtime evidence: it observes with
fentry/fexit probes, turns raw kernel signals into validated evidence, and
hands that evidence to Polku for delivery to Vartio (which interprets and writes
to Ahti). Jalki must not write to Ahti directly and must not become a parallel
datastore. See `docs/jalki/adr/0002-evidence-through-polku-to-vartio.md`.

## Agentic Dev Lane

Use this lane for non-trivial agent work:

1. Human idea becomes a structured issue or task contract.
2. Agent writes an implementation plan before editing.
3. Work happens on a branch scoped to one ticket.
4. A different model/tool reviews the diff.
5. CI and targeted tests judge the change.
6. A human merges.
7. New repeated rules become updates to `AGENTS.md` and `CLAUDE.md`.

Agents do not own truth. Issues, PRs, tests, specs, design docs, and reviewed
commits own truth.

## Product Boundaries

- Jalki emits kernel and runtime evidence; it does not own durable evidence
  storage.
- Evidence reaches Ahti via Polku → Vartio; Vartio is the writer. Jalki must not
  write to Ahti directly and must not keep a parallel datastore.
- Jalki may keep local node state only to observe, attach probes, buffer events
  briefly, and report status.
- Lähde and Vartio interpret and correlate; do not move broad observability or
  causal-attribution product logic into Jalki.
- Keep internal Finnish architecture names out of buyer-facing benefit language
  unless the task explicitly asks for positioning work.

## Mental Model

Jalki has four main layers:

- `jalki-common`: shared `no_std` kernel/userspace event structs and ABI tests.
- `jalki-ebpf`: separate eBPF build target; not a workspace member.
- `jalki`: userspace daemon, CLI, probe registry, loader, reader, IPC, metrics,
  and embedded knowledge base.
- `jalki-sdk-meta`: source of truth for SDK types, protocol framing, and SDK
  generation.

The local API speaks over `/run/jalki/jalki.sock` using the binary frame
protocol defined in `jalki-sdk-meta/src/protocol.rs`. Do not hardcode wire
message bytes elsewhere.

## Build And Test

Build order matters on Linux:

```bash
cargo run -p xtask -- build-ebpf
cargo build -p jalki
```

Full Linux checks:

```bash
cargo check --workspace
cargo test --workspace
cargo test --manifest-path eval/oracle/Cargo.toml
```

On macOS, `aya`-dependent crates do not compile. Use focused checks:

```bash
cargo check -p jalki-common
cargo check -p jalki-sdk-meta
cargo check -p xtask
cargo test -p jalki-common
cargo test -p jalki-sdk-meta
cargo test --manifest-path eval/oracle/Cargo.toml
cargo clippy -p jalki-common -p jalki-sdk-meta
```

The daemon needs Linux and root or `CAP_BPF` plus `CAP_PERFMON`.

## Change Rules

- Keep `jalki-common` `no_std`.
- Preserve `#[repr(C)]` event layouts unless deliberately changing the BPF ABI.
- Add or update struct size tests for every event ABI change.
- Do not use `.unwrap()` in userspace code; propagate or handle errors.
- Do not use `println!` in library code; use `tracing`.
- Use `thiserror` for library errors and `anyhow` for binary entry points.
- Document why each eBPF `unsafe` block is correct.
- Do not change public APIs without an explicit migration note.
- Do not add production dependencies without approval.
- Never log secrets, tokens, personal data, raw payloads, SQL text, or full
  secret-bearing config structures.
- Every behavior change needs a test or a documented reason existing coverage is
  sufficient.
- Keep PRs small. Prefer one concern per PR.
- If a task reveals a missing repeated rule, propose an update here.
- Never hand-edit files marked `# GENERATED by jalki-sdk-meta`; edit
  `jalki-sdk-meta/src/` and regenerate.

## SDK Workflow

When SDK types or protocol definitions change:

```bash
cargo run -p jalki-sdk-meta -- --lang python --out jalki-sdk-python/src/jalki/
cargo test -p jalki-sdk-meta
cargo test --manifest-path eval/oracle/Cargo.toml
```

`jalki-sdk-python` is not a workspace member. Its generated files should match
the meta crate, not hand-written assumptions.

## Adding A Probe

Add probes in this order:

1. Define the event struct in `jalki-common/src/events.rs`, with `#[repr(C)]`,
   padding/alignment, size tests, and `aya::Pod` behind the `userspace` feature.
2. Add the eBPF program in `jalki-ebpf/src/`, including ring buffer map,
   `PID_FILTER`, and `main.rs` wiring.
3. Implement `Probe` in `jalki/src/probes/`. `program_name()` must exactly match
   the eBPF `#[fentry]` or `#[fexit]` function name.
4. Add a certain, validated knowledge entry under `knowledge/`.

Use `fexit` when return values or errno matter. Use `fentry` when the question is
only whether a function was reached.

## Oracle And Specs

`eval/oracle/` is a standalone contract test suite. It must validate public
behavior from files and specs, not import implementation code.

When an oracle case fails, fix the system or data. Do not weaken oracle cases as
a side effect of implementation work.

`specs/` contains Luotain-compatible requirement specs. Keep specs and oracle
expectations aligned when behavior changes.

## Debugging Notes

- Userspace can compile without the eBPF object, but the daemon will fail at
  runtime if it is missing.
- Jalki self-filters its own PID in kernel space; this is expected behavior.
- Some kernel fields have known caveats, including Cilium-managed destination IP
  reads, `tcp_close` source port clearing, and hardcoded `tcp_sock` offsets.
- Verify kernel layout through BTF tools such as `pahole` when touching offsets.

## Git Hygiene

format-only churn unless it is needed for a checked gate. Prefer the repo's
existing trait and crate boundaries over introducing new abstractions.
